# 代码审查清单

## 审查前准备

1. 拉取最新代码，确保能本地构建通过
2. 理解 PR 的背景和目标（阅读 PR 描述和关联 issue）
3. 先看整体改动范围，再逐文件审查

## 审查维度

### 🔴 阻断项（Must Fix）

- [ ] 安全漏洞（SQL 注入、XSS、CSRF、敏感信息泄露）
- [ ] 数据丢失风险（错误的 DELETE/UPDATE、缺少事务）
- [ ] 生产环境崩溃风险（未处理的异常、空指针）
- [ ] API 契约破坏（Breaking Change 未标注）
- [ ] 硬编码密钥或凭证

### 🟡 建议项（Should Fix）

- [ ] 命名不符合规范
- [ ] 缺少必要的错误处理
- [ ] 缺乏类型定义（使用 any）
- [ ] 函数过长或职责不单一
- [ ] 重复代码可抽取
- [ ] 缺少必要的测试
- [ ] 性能问题（N+1 查询、不必要的重渲染）
- [ ] 不必要的依赖引入

### 🟢 建议项（Nice to Have）

- [ ] 更好的命名建议
- [ ] 代码风格改进
- [ ] 文档补充
- [ ] 测试覆盖提升

## 审查评论模板

```markdown
**[MUST]** 安全问题：此处存在 SQL 注入风险，请使用参数化查询。
**[SHOULD]** 建议提取为独立函数 `calculateTax()`，当前函数已超过 80 行。
**[NIT]** 建议将变量名 `d` 改为 `discount` 以提高可读性。
**[QUESTION]** 这里选择 setTimeout 而非 debounce 的原因是什么？
```

## AI 辅助审查要点

当使用 AI 进行代码审查时，关注以下额外维度：

- 生成代码是否包含幻觉（不存在的 API、错误的方法签名）
- 是否引入了不必要的复杂度
- 错误处理是否健壮
- 是否遵循项目既有的代码风格和模式
