# 代码审查规范

## 审查原则

1. **代码是对的，人没有对错** — 审查代码，不评判人
2. **建设性为先** — 每个问题附带改进建议
3. **把握重点** — 安全和正确性 > 性能 > 风格
4. **及时响应** — 首次审查在 24 小时内完成

## 审查清单

### 🔴 阻断项（MUST FIX）

必须修复后才能合并：

**安全**

- [ ] SQL 注入（未使用参数化查询）
- [ ] XSS（用户输入未转义直接渲染）
- [ ] CSRF（缺少 token 验证）
- [ ] 敏感信息泄露（密钥、token、密码硬编码或日志输出）
- [ ] 不安全的反序列化
- [ ] 路径遍历（用户输入直接拼接文件路径）
- [ ] 权限校验缺失或绕过

**正确性**

- [ ] 空指针 / undefined 访问
- [ ] 数据丢失风险（错误的 DELETE/UPDATE、缺少事务）
- [ ] 竞态条件（并发场景下共享状态无保护）
- [ ] 资源泄漏（未关闭连接、文件句柄、定时器）
- [ ] API 契约破坏（Breaking Change 未标注版本）
- [ ] 死循环 / 无限递归

### 🟡 重点关注（SHOULD FIX）

强烈建议修复：

**设计与结构**

- [ ] 函数超过 50 行或职责不单一
- [ ] 类/模块职责过重（God Object）
- [ ] 重复代码可抽取（DRY 原则）
- [ ] 过度抽象（YAGNI — 不需要的功能不要提前设计）
- [ ] 循环依赖
- [ ] 不合理的继承关系（优先组合而非继承）

**类型与错误处理**

- [ ] 使用 `any` 类型（TypeScript）
- [ ] 缺少必要的错误处理
- [ ] 吞掉异常（catch 后不处理）
- [ ] 错误类型过于宽泛（通用 `Error`）

**性能**

- [ ] N+1 查询
- [ ] 不必要的重渲染（React）
- [ ] 缺少分页的大数据查询
- [ ] 同步阻塞操作在主线程
- [ ] 缺少索引的频繁查询

**测试**

- [ ] 核心业务逻辑缺少单元测试
- [ ] 新增 API 缺少集成测试
- [ ] 测试覆盖了正常路径但缺少边界情况

### 🟢 建议改进（NICE TO HAVE）

可选优化，不阻塞合并：

- [ ] 命名可以更清晰
- [ ] 代码风格可优化
- [ ] 缺少注释（复杂逻辑的"为什么"）
- [ ] 文档可补充
- [ ] 测试覆盖率可提升
- [ ] 可使用更简洁的语法

## 审查评论规范

### 前缀标签

每条审查评论必须使用前缀标签表明严重程度：

```text
[MUST]     — 阻断项，必须修复
[SHOULD]   — 建议项，强烈建议修复
[NIT]      — 微小建议，可选
[QUESTION] — 疑问，需要作者解释
[PRAISE]   — 表扬，好的实践值得肯定
```

### 评论模板

```markdown
**[MUST]** 安全漏洞：此处将用户输入直接拼接到 SQL 查询中，存在 SQL 注入风险。
建议使用参数化查询：
​```sql
SELECT * FROM users WHERE id = ?
​```

**[SHOULD]** 函数 `processOrder()` 已超过 80 行，建议拆分：
- `validateOrder()` — 校验订单数据
- `calculateTotal()` — 计算总价
- `saveOrder()` — 持久化

**[NIT]** 变量名 `d` 建议改为 `discountRate`，提高可读性。

**[QUESTION]** 这里选择 `setTimeout` 而非 `debounce` 的原因是什么？
是否需要考虑组件卸载时清除定时器？

**[PRAISE]** 这个 `useRetry` Hook 封装得很好，复用性强，错误处理完整。
```

### 评论原则

- 指出问题的同时给出**具体的修复建议**
- 引用相关文档或最佳实践链接
- 对于复杂问题提供代码示例
- 一个评论聚焦一个问题，不要混杂

## 按语言/框架的审查要点

### TypeScript / JavaScript

- 是否启用 `strict` 模式
- 是否使用 `any`（应使用 `unknown` + 类型守卫）
- `async/await` 是否正确处理错误
- `useEffect` 依赖数组是否完整（React）
- 是否有内存泄漏风险（事件监听器、订阅未清理）

### Python

- type hints 是否完整
- 是否使用 Pydantic 校验入参
- 异常处理是否过于宽泛（`except Exception`）
- 是否正确使用 `async/await`
- 上下文管理器是否合理使用（`with` 语句）

### Go

- 错误是否被正确处理（不要 `_ = err`）
- goroutine 是否有泄漏风险
- 接口是否定义在消费方
- `context.Context` 是否正确传递
- defer 是否在正确位置

### SQL / 数据库

- 是否使用参数化查询
- 大量数据操作是否在事务中
- 是否有缺少索引的慢查询
- 迁移脚本是否可回滚
- 是否考虑了并发写入冲突

## AI 辅助审查补充

当使用 AI 进行代码审查时，额外关注：

- 生成代码是否包含**幻觉**（不存在的 API、错误的方法签名）
- 是否引入了**不必要的复杂度**（过度使用设计模式）
- 是否遵循项目**既有的代码风格**和文件组织模式
- 错误处理是否对所有边界情况都健壮
- 是否有**版权问题**（直接复制开源项目代码但 License 不兼容）

## 审查结果状态

| 状态 | 含义 | 后续动作 |
| ---- | ---- | -------- |
| ✅ APPROVED | 审查通过 | 可合并 |
| 🔄 NEEDS_REVISION | 需要修改 | 作者修改后重新请求审查 |
| ❌ REJECTED | 方案有根本问题 | 需重新讨论方案 |
